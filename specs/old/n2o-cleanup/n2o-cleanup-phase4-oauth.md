### Scoping model

All data in the remote API is scoped by **(user_id, project_id)**. A project is tied to a GitHub repo (set during `n2o init`). Each developer's tasks, events, and sprints are stored under their user ID within that project.

```
Organization (better-auth org)
 └─ Project (= GitHub repo, e.g. github.com/org/repo)
     └─ User scope (= developer)
         ├─ Tasks (created by this user's agents)
         ├─ Events (generated by this user's n2o commands)
         └─ Sprints (created by this user's pm-agent)
```

**v1: single-user per project.** Each developer has their own task set. `n2o sync` pushes/pulls only that developer's data. The API stores events tagged with `(user_id, project_id)` and filters on pull.

**Why scope by user**: A developer's pm-agent generates tasks tailored to that developer — based on their skills, their sprint plan, their spec. These tasks are meaningless to another developer. The remote API is a backup + cross-device sync layer for one developer's workflow, not a shared task board.

### Data flow

1. **CLI → API → Postgres (write path)**: CLI sends events scoped by `(user_id, project_id)`. API validates bearer token, INSERTs into Postgres. Events are append-only, scoped rows.
2. **Postgres → CDC → Tinybird (pipeline)**: CDC captures new events from Postgres WAL via logical replication. A lightweight consumer (Estuary Flow or Redpanda Connect) forwards them to Tinybird's Events API as NDJSON. No Kafka needed at this scale.
3. **Tinybird → Web app (read path)**: Frontend calls Tinybird published endpoints directly with JWT for dashboards. Tinybird handles all analytical queries. Row-level security via `user_id` / `org_id` in JWT.
4. **API → CLI (pull path)**: CLI pulls events filtered by `(user_id, project_id)` from the API. CLI materializes pulled events into local `.pm/workflow.db`.

### Why this separation

**Nothing reads analytics from Postgres.** Postgres is write-optimized only for the event log. All analytical reads go through Tinybird. This prevents OLTP + OLAP contention on the same database as scale grows.

**Postgres handles what it's good at**: concurrent writes, ACID transactions, user/project scoping, auth/sessions (better-auth).

**Tinybird handles what it's good at**: columnar scans across all users' data, materialized views updated on every insert, published REST endpoints for dashboards, JWT with row-level security.

### Data stores

| Store | Technology | Role | Reads | Writes |
|-------|-----------|------|-------|--------|
| `.pm/workflow.db` | SQLite (local) | Task state + event log + telemetry (single DB) | CLI reads constantly | `n2o task *` writes, lazy transcript parsing |
| **Postgres** | Managed | Canonical event log + auth/users | API reads for pull, conflict checks | API writes events, auth |
| **Tinybird** | Managed ClickHouse | Analytics + dashboards | Frontend reads via JWT | CDC pipeline writes |

> **Note**: Phase 5 consolidated the original three-DB split into a single `.pm/workflow.db`. The `event` table lives alongside task state for atomic transactions. Telemetry tables (`telemetry`, `transcript`, `message`, `tool`) also live in the same DB. All tables use singular names.

### Postgres write optimization

The events table is append-only, typed columns (not JSON), and write-optimized:

- **Typed columns, not JSON** — API destructures the CLI's JSON payload into typed Postgres columns on INSERT. CDC replicates typed rows to ClickHouse with no transformation. See `workflow-events.md` for full schema.
- **No foreign keys** on the events table — sprint/task/user stored as values, not FK references
- **BRIN index on `timestamp`** — 500x smaller than B-tree, perfect for append-only time-series
- **Index on `(user_id, project_id)`** — scoped queries for pull
- **Time-based partitioning** (daily via pg_partman) — retention cleanup is `DROP PARTITION` (instant, no vacuum)
- **30-day retention** — old events are in Tinybird; Postgres is just the write buffer
- **Tables are LOGGED** (not UNLOGGED) — required for CDC/logical replication to work

### CDC pipeline: Postgres → Tinybird

No Kafka needed at this scale (thousands to low millions of events/day).

**Simplest option: Estuary Flow**
- Managed CDC service. Reads Postgres logical replication, delivers to Tinybird via Kafka-compatible interface (Dekaf).
- Exactly-once delivery. Automatic schema evolution. Monitors replication lag.
- Tinybird's native Kafka connector consumes from Estuary's Dekaf endpoint.

**Alternative: Redpanda Connect**
- Lightweight CDC bridge. Tinybird has a documented guide for Postgres CDC via Redpanda Connect.
- Self-hosted, single binary, lower cost.

**Alternative: Custom Go consumer**
- Read Postgres logical replication stream (`pgoutput` plugin), POST to Tinybird Events API via HTTP.
- Maximum control, minimal dependencies. ~200 lines of Go.

**Key operational concern: replication slot lag.** If the CDC consumer falls behind, Postgres retains WAL files, which can fill disk. Monitor `pg_replication_slots`, set `max_slot_wal_keep_size` to cap WAL retention.

### Why not skip Postgres and write directly to Tinybird?

- **No conflict detection**: Tinybird/ClickHouse is append-only. You can't reject a double-claim with a unique constraint. You'd have to query Tinybird before every write to check, which defeats the purpose.
- **No ACID transactions**: Can't atomically validate + insert.
- **No pull source for CLI**: The CLI needs to pull current task state. Querying Tinybird for this works but adds latency and couples the CLI to Tinybird's availability.
- **Dual-write problem**: Writing to both Postgres and Tinybird directly creates consistency issues. CDC from a single source (Postgres) avoids this.

## What the CLI sends (push)

Events posted to your API server (not directly to Tinybird):

```
POST /api/projects/:id/events
Authorization: Bearer <session_token>
Content-Type: application/json

{
  "events": [
    {"event_id":"uuid","event_type":"task.claimed","timestamp":"...","payload":{...}},

```

API validates → INSERTs to Postgres → returns `{ accepted: N, rejected: [...] }`.
CDC picks up new rows from Postgres WAL → forwards to Tinybird asynchronously.

## Auth: better-auth with device flow

The web app uses [better-auth](https://better-auth.com) for authentication. Better-auth has a first-party `deviceAuthorization` plugin that implements RFC 8628 natively — this is the exact flow needed for CLI → web app auth.

Better-auth is the only self-hosted JS auth library with native device flow, API keys, bearer tokens, and organizations — all as first-party plugins.

### Plugins to enable on the web app

```typescript
import { betterAuth } from "better-auth";
import { deviceAuthorization } from "better-auth/plugins";
import { bearer } from "better-auth/plugins";
import { organization } from "better-auth/plugins";
import { apiKey } from "@better-auth/api-key"; // optional, for long-lived keys

export const auth = betterAuth({
  database: postgres,  // Postgres adapter
  plugins: [
    deviceAuthorization({
      verificationUri: "/device",    // page where user enters code
      expiresIn: 1800,               // 30 min to approve
      interval: 5,                   // CLI polls every 5s
    }),
    bearer(),                        // enables Authorization: Bearer <token>
    organization({
      teams: { enabled: true },      // org → team → member hierarchy
    }),
    apiKey(),                          // long-lived API keys for CI/runners (always project-scoped)
  ],
});
```

This adds a `deviceCode` table to Postgres and exposes the device flow endpoints automatically.

### Login flow

```
$ n2o login
→ CLI sends POST /api/auth/device-authorization/authorize
    { client_id: "n2o-cli" }
→ App returns:
    { device_code, user_code: "ABCD-EFGH", verification_uri, interval, expires_in }

→ User logs in on the web app (if not already), enters code, clicks Approve
```

## What changes in the CLI

1. **`n2o login`** — OAuth device flow against the app's API. Stores tokens in `~/.n2o/credentials.json`.
2. **`n2o logout`** — Clears stored credentials.
3. **`n2o status`** — Shows auth state, sync state (pending events, last sync, online/offline). Phase 6 extends this to also show connected account status (Claude, GitHub) fetched from the API.
4. **`n2o sync`** — Flush pending events to remote, pull remote state into local DB.
5. **`n2o apikey create --name "CI runner"`** — Generate a long-lived API key for the current project. Requires login. Always scoped to the project in `.pm/config.json` — no scope flag needed. Used for CI runners, async jobs, automation.
6. **`n2o apikey list`** — List active API keys for this project (name, created date, last used). Does not show the key value after creation.
7. **`n2o apikey revoke --name "CI runner"`** — Revoke an API key by name.
8. **Lazy sync** — first `n2o task *` call in a session flushes pending events + pulls if authenticated. No hook needed.
9. **Local-first mutations** — All task mutations (via `n2o task *`, see phase 5) update local DB first, then push events to remote as best-effort. Local state is always correct even if push fails.

## Sync model: Event log, not snapshots

The naive approach (periodically push local DB state) has serious problems:
- **Tampering**: User edits SQLite directly → remote gets fake data
- **Deletion**: Local DB deleted → all unpushed changes lost, no recovery
- **Partial sync**: Crash mid-push → remote has half the data
- **Staleness**: Periodic sync means remote is always behind

Instead: **the remote is the source of truth.** The CLI sends immutable events, not state snapshots.

### How it works

Every task mutation in the CLI produces an **event** — an immutable record of what happened.

```
┌─ CLI mutation ──────────────────────────────────────────────┐
│                                                             │
│  1. Write event to local event log (SQLite WAL table)       │
│  2. Send event to remote API                                │
│     ├─ Online: API accepts → mark event as synced           │
│     └─ Offline: event stays in local log, retry later       │
│  3. Update local workflow.db (materialized from events)        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Remote rebuilds state from events.** Postgres stores the canonical event log. Current task state is derived from events. This means:
- Remote is always accurate (events are the source of truth)
- Local DB can be rebuilt anytime from remote events (`n2o sync --rebuild`)
- Tampering with local SQLite doesn't matter — events already sent can't be unsent
- Partial sync is safe — each event is independent and idempotent

### Event types

| Event | Payload | When |
|-------|---------|------|
| `task.created` | Full task row | PM seeds tasks |
| `task.claimed` | sprint, task_num, owner, session_id | TDD agent claims task |
| `task.status_changed` | sprint, task_num, old_status, new_status | RED→GREEN, etc. |
| `task.audit_completed` | sprint, task_num, testing_posture, pattern_audited | After AUDIT phase |
| `task.committed` | sprint, task_num, commit_hash, lines_added/removed | After COMMIT phase |
| `task.verified` | sprint, task_num, verified_by | PM verifies |
| `task.blocked` | sprint, task_num, blocked_reason | Task blocked |
| `task.unblocked` | sprint, task_num | Block resolved |
| `dep.created` | sprint, task_num, depends_on_sprint, depends_on_task | Dependency added |
| `sprint.created` | sprint name, spec | New sprint seeded |

### Event schema (local SQLite)

```sql
CREATE TABLE IF NOT EXISTS event (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id TEXT NOT NULL UNIQUE,      -- UUID, generated client-side
    event_type TEXT NOT NULL,           -- task.claimed, task.status_changed, etc.
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,
    user_id TEXT NOT NULL,              -- from auth token
    project_id TEXT NOT NULL,
    payload TEXT NOT NULL,              -- JSON blob
    synced_at DATETIME,                 -- NULL = not yet synced
    sync_attempts INTEGER DEFAULT 0,
    sync_error TEXT
);

CREATE INDEX IF NOT EXISTS idx_event_unsynced
    ON event(synced_at) WHERE synced_at IS NULL;
```

### Write path (local-first, always)

All mutations go through `n2o task *` commands (phase 5). The write path is identical regardless of auth/connectivity state:

```
n2o task status --sprint auth --task 3 --status green
  ↓
1. Validate state transition (current status must be 'red')
2. BEGIN TRANSACTION (local .pm/workflow.db)
   a. UPDATE tasks SET status = 'green' WHERE sprint = ? AND task_num = ?
   b. INSERT INTO event (event_id, event_type, user_id, project_id, payload, ...)
3. COMMIT
4. If authenticated + online:
   POST /api/projects/:id/events { event }
   ├─ 200 OK → UPDATE event SET synced_at = NOW()
   └─ 5xx / timeout → leave unsynced, retry on next sync
5. Print: ✓ auth/3 → green (synced|queued|local only)
```

**Local state is always updated first.** The remote push is best-effort. If the user is logged out, no push is attempted — events are written with `user_id = NULL` and don't sync.

### Claim conflict model

Task claiming is **single-developer, multi-agent**: one person runs 8-10 parallel Claude Code terminals, each agent claims a different task. All claims happen against the same local `.pm/workflow.db` — SQLite's WAL mode + atomic `WHERE owner IS NULL` handles concurrency natively. No remote conflict detection needed.

The remote API receives claim events as informational records, not as lock requests. If a future multi-user mode is added, claim conflict detection would need to move to the API layer. For now, all claims are local-only atomic operations.

### Read path (pull)

```
n2o sync
  ↓
1. GET /api/projects/:id/events?user=me&since=<last_seen_event_id>
2. Response: { events: [...], cursor: "last_event_id" }
3. For each event:
   → Apply to local workflow.db (idempotent — skip if event_id already seen)
4. Store cursor in .pm/config.json
```

Pull is filtered by the authenticated user — each developer only pulls their own events.

### Rebuilding local DB from remote

If local `.pm/workflow.db` is deleted or corrupted:

```
$ n2o sync --rebuild
  ↓
1. GET /api/projects/:id/events?user=me&since=0 (all my events)
2. Create fresh workflow.db from schema
3. Replay all events in order → materialized task state
4. Done — local DB matches remote exactly
```

### Edge cases

| Scenario | What happens |
|----------|-------------|
| **Local DB deleted** | `n2o sync --rebuild` replays all events from remote. No data loss. |
| **Local DB tampered with** | Doesn't matter — remote state is derived from events, not from local DB. Tampered local state gets overwritten on next pull. |
| **Crash during event send** | Event is in local log (synced_at = NULL). Retried on next sync. Remote either has it (idempotent, no-op) or doesn't (gets it now). |
| **Crash during pull** | Cursor not updated. Next pull re-fetches same events. Idempotent — safe to replay. |
| **Network timeout mid-push** | CLI doesn't know if remote got it. On retry, remote sees duplicate event_id → returns 200 (idempotent). |
| **Dev offline for a week** | All events queued locally. On reconnect, flushed in order. Remote stores them under this user's scope. |
| **Second machine** | `n2o sync --rebuild` on new machine pulls all events for this user + project → identical local state. |

### Tamper resistance

The event log model makes tampering much harder than snapshot sync:

1. **Events are immutable and append-only.** You can't un-send an event. Editing local SQLite doesn't change what the remote already received.
2. **Remote derives state from its own event log.** It never trusts a "here's my current task state" push from the client.
3. **Event signatures (optional hardening):** Each event could include `HMAC(event_payload, user_secret)` — remote validates the signature. Requires server-issued per-user signing key.
4. **Audit trail is permanent.** Every state change has a who/when/what record in Postgres. Can't be retroactively altered from the client.

## Progressive rollout

| Stage | Behavior |
|-------|----------|
| **No account** | Everything works locally. Events written to local log but never sent. Same as phase 3 from user's perspective. |

```
GET    /api/auth/get-session                          — Validate bearer token, return user/org
POST   /api/auth/api-key/create                       — Create project-scoped API key
GET    /api/auth/api-key/list                         — List API keys for user
POST   /api/auth/api-key/revoke                       — Revoke an API key
```

API keys authenticate the same way as bearer tokens — the API validates them via better-auth's `apiKey` plugin. The key is scoped to the project (stored as metadata on the key). Events pushed with an API key are attributed to the user who created the key.

```
  Request:  { events: [{ event_id, event_type, timestamp, payload }, ...] }
  Response: { accepted: N, rejected: [{ event_id, reason }] }

            INSERTs to Postgres scoped by (user_id, project_id).
```

## Verification

### Auth
- `n2o login` opens browser, completes device flow, stores token
- `n2o logout` clears credentials
- `n2o status` shows authenticated user, org, pending events, last sync

### Sync correctness
- Developer pushes events from laptop → pulls on second machine → sees same task state
- Events are scoped by (user_id, project_id) — each developer's data is isolated
- Task claims are local-only atomic operations — no remote conflict detection needed
- With no account: all commands work identically to phase 3 (events written locally, never sent)

### Resilience
- Delete `.pm/workflow.db` → `n2o sync --rebuild` → local DB rebuilt from remote events
- Kill CLI mid-sync → restart → unsynced events in local log → flushed on next sync
- Go offline for a day → come back → `n2o sync` flushes all queued events in order
- Network timeout during event push → retry → remote deduplicates by event_id (idempotent)
- Tamper with local workflow.db → next pull overwrites tampered state with remote-derived state

### Data isolation
- Telemetry tables (`telemetry`, `transcript`, `message`, `tool`) never leave the machine
- Only `event` rows with `user_id IS NOT NULL` are pushed to remote
- Event push sends the event payload, never raw task state
- Remote state is derived from events, never from a client-provided state snapshot

## Future: Multi-user extension

v1 is single-user per project — each developer's agent-level tasks are isolated in their own `(user_id, project_id)` scope. The `task` table is an agent work queue (15-60 minute granular items), not a shared project board.

Multi-user collaboration operates at a **higher level of abstraction** — objectives, epics, or deliverables that a project lead defines and assigns to developers. Each developer's agents then break their assigned work into the granular tasks they execute locally.

### Architecture: two layers

```
App layer (shared, multi-user)          CLI layer (per-developer, local)
┌─────────────────────────────┐        ┌──────────────────────────────┐
│ objective / epic / milestone│        │ task table (agent work items)│
│                             │        │                              │
│ Created by project lead     │───────▸│ pm-agent breaks objective    │
│ Assigned to developers      │        │ into 15-60 min tasks         │
│ Visible to whole team       │        │ tdd-agent executes them      │
│ Managed in web app          │        │ Local to this developer      │
│ Stored in Postgres          │        │ Stored in .pm/workflow.db    │
└─────────────────────────────┘        └──────────────────────────────┘
```

The app layer is a **new set of tables in Postgres** (objective, assignment, milestone — whatever the right nouns are). It does not modify the `task` table, the `event` table, or any CLI commands. The `task` table stays per-developer, agent-scoped, local-first.

### What doesn't change

- **`task` table** — still per-developer agent work items, never shared across users
- **`event` table** — still scoped by `(user_id, project_id)`, still local-first
- **`n2o task *` commands** — same interface, same local-only claiming
- **Event-sourced sync** — each developer's events still push/pull independently
- **Adapter interface** — coding tool abstraction is user-agnostic

### What the app layer adds

| Concern | Where it lives | Who interacts |
|---------|---------------|---------------|
| **Objectives/epics** | Postgres (app DB) | Project lead via web app |
| **Assignments** | Postgres (app DB) | Project lead assigns objectives to developers |
| **Progress rollup** | Postgres views / Tinybird | Dashboard aggregates each developer's task completion against their assigned objectives |
| **Cross-developer visibility** | Web app dashboard | Project lead sees all developers' progress |

Design details deferred. The key constraint is: the `task` table, `event` table, CLI task commands, and local DB structure do not change.

## Appendix: Local database schema (`.pm/workflow.db`)

Single SQLite database. WAL mode. Auto-migrates on open. All tables use singular names.

### Tables that sync (via event → remote API)

```sql
-- Task state (primary key: sprint + task_num)
task (sprint, task_num, title, status, owner, type, done_when,
      estimated_minutes, complexity, testing_posture, commit_hash, ...)

-- Dependency graph
dep (sprint, task_num, depends_on_sprint, depends_on_task)

-- Event log: every n2o task * command generates an event here
-- Unsynced events (synced_at IS NULL) flush on next n2o sync or
-- lazily on first n2o task * call in a new session
event (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id TEXT NOT NULL UNIQUE,       -- UUID, generated client-side
    event_type TEXT NOT NULL,            -- task.claimed, task.status_changed, etc.
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,
    user_id TEXT,                        -- from auth token (NULL if logged out)
    project_id TEXT,                     -- from .pm/config.json
    payload TEXT NOT NULL,               -- JSON blob
    synced_at DATETIME,                  -- NULL = not yet synced
    sync_attempts INTEGER DEFAULT 0,
    sync_error TEXT
)

-- Developer profiles (pulled from remote)
developer (name, full_name, role, ...)
```

### Tables that stay local

```sql
-- Telemetry: tool calls, phase transitions, subagent spawns
-- Populated lazily from JSONL transcripts + by n2o phase enter
telemetry (id, timestamp, session_id, sprint, task_num, event_type,
           tool_name, skill_name, phase, agent_type, input_tokens, ...)

-- Session transcripts: one row per JSONL file
-- Populated lazily (parsed on first n2o task/stats call if un-indexed JSONL exists)
transcript (session_id, file_path, message_count, total_input_tokens,
            total_output_tokens, estimated_cost_usd, model, started_at, ...)

-- Full message content (privacy-sensitive)
message (session_id, message_index, role, content, timestamp, ...)

-- Full tool call inputs
tool (session_id, message_index, tool_index, tool_name, input, ...)

-- Pre-aggregated daily stats from coding tool (e.g., stats-cache.json)
stat (date, adapter, message_count, session_count, tool_call_count,
      tokens_by_model)

-- Developer context snapshots
context (developer, concurrent_sessions, hour_of_day, alertness, ...)

-- Skill version tracking
skill (skill_name, version, framework_version, ...)

-- Migration tracking
migration (name, applied_at, framework_version, checksum)
```
