# Workflow Event Catalog
> Every event the N2O system produces, where it lives, and whether it leaves the machine.

## Event Destinations

All local data lives in a single `.pm/workflow.db` (SQLite). Events that sync to remote are in the `event` table.

| Destination | Store | What goes here |
|-------------|-------|---------------|
| **Local** | `.pm/workflow.db` (SQLite) | All task state, event log, telemetry, transcripts — single DB |
| **Remote** | Postgres (via API push) | Copy of `event` table rows with `user_id IS NOT NULL`, scoped by `(user_id, project_id)` |
| **Analytics** | Tinybird (via CDC from Postgres) | Same events, columnar format for dashboards |

## 1. Task Lifecycle Events (Sync)

These are the event-sourced mutations from phase 4. Generated by `n2o task *` commands. They flow: CLI → `event` table (local) → API → Postgres → CDC → ClickHouse.

| Event | Payload | When | Source |
|-------|---------|------|--------|
| `task.created` | Full task row | PM seeds tasks | pm-agent |
| `task.claimed` | sprint, task_num, owner, session_id | TDD agent claims task | tdd-agent |
| `task.released` | sprint, task_num, previous_owner | Task unclaimed (timeout, manual) | CLI / tdd-agent |
| `task.status_changed` | sprint, task_num, old_status, new_status | RED→GREEN, pending→red, etc. | tdd-agent |
| `task.audit_completed` | sprint, task_num, testing_posture, pattern_audited, grades | After AUDIT phase | tdd-agent |
| `task.committed` | sprint, task_num, commit_hash, lines_added, lines_removed | After COMMIT phase | tdd-agent |
| `task.verified` | sprint, task_num, verified_by | PM verifies | pm-agent / CLI |
| `task.blocked` | sprint, task_num, blocked_reason | Task blocked | tdd-agent / bug-workflow |
| `task.unblocked` | sprint, task_num | Block resolved | pm-agent / CLI |
| `task.reverted` | sprint, task_num, old_status, new_status | Status went backward (green→red) | tdd-agent |
| `task.priority_changed` | sprint, task_num, old_priority, new_priority | Reordering | pm-agent |
| `task.estimated` | sprint, task_num, estimated_minutes, complexity | Estimate assigned or revised | pm-agent |

## Remote schema

### Pipeline

```
CLI (JSON payload) → API (maps to typed columns) → Postgres (typed) → CDC → ClickHouse (typed)
```

The CLI sends events as JSON (`{ event_type, payload: {...} }`). The **API is the mapping layer** — it destructures the payload into typed Postgres columns on INSERT. CDC is a dumb pipe that replicates typed rows to ClickHouse. No JSON parsing in the analytics layer.

### Postgres event table (typed columns, not JSON)

```sql
CREATE TABLE event (
    event_id UUID PRIMARY KEY,
    event_type TEXT NOT NULL,           -- LowCardinality in ClickHouse
    timestamp TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    user_id UUID NOT NULL,
    project_id UUID NOT NULL,

    -- Common fields (nullable — not every event type uses all of them)
    sprint TEXT,
    task_num INTEGER,
    session_id TEXT,

    -- Task fields
    old_status TEXT,
    new_status TEXT,
    owner TEXT,
    blocked_reason TEXT,
    testing_posture TEXT,
    pattern_audited BOOLEAN,
    commit_hash TEXT,
    lines_added INTEGER,
    lines_removed INTEGER,
    title TEXT,                          -- for task.created
    task_type TEXT,                      -- for task.created
    done_when TEXT,                      -- for task.created

    -- Dependency fields
    depends_on_sprint TEXT,
    depends_on_task INTEGER,

    -- Sprint fields
    goal TEXT,
    deadline TIMESTAMPTZ,

    -- Phase fields
    phase TEXT,
    skill TEXT,

    -- Metadata (anything that doesn't warrant its own column)
    meta JSONB                           -- escape hatch, not the default
);

CREATE INDEX idx_event_user_project ON event(user_id, project_id, timestamp);
CREATE INDEX idx_event_type ON event(event_type, timestamp);
```

Wide table with nullable columns. Each event type populates only its relevant columns — the rest are NULL. This is how ClickHouse works best: wide, sparse, columnar. Postgres handles it fine too (NULLs are cheap in both).

The `meta` JSONB column is an escape hatch for truly unstructured data (e.g., pattern audit notes, skill update notes). Not the primary query path.

### ClickHouse table (mirrors Postgres, via CDC)

```sql
CREATE TABLE event (
    event_id UUID,
    event_type LowCardinality(String),
    timestamp DateTime64(3),
    user_id UUID,
    project_id UUID,
    sprint Nullable(String),
    task_num Nullable(UInt32),
    session_id Nullable(String),
    old_status Nullable(String),
    new_status Nullable(String),
    owner Nullable(String),
    blocked_reason Nullable(String),
    testing_posture Nullable(String),
    pattern_audited Nullable(UInt8),
    commit_hash Nullable(String),
    lines_added Nullable(Int32),
    lines_removed Nullable(Int32),
    title Nullable(String),
    task_type Nullable(String),
    phase Nullable(String),
    skill Nullable(String),
    depends_on_sprint Nullable(String),
    depends_on_task Nullable(UInt32),
    goal Nullable(String),
    deadline Nullable(DateTime64(3)),
    meta Nullable(String)
) ENGINE = MergeTree()
ORDER BY (project_id, user_id, event_type, timestamp)
PARTITION BY toYYYYMM(timestamp);
```

CDC replicates rows 1:1. No transformation. ClickHouse compresses the sparse NULLs efficiently.

### API mapping responsibility

The API's event ingestion endpoint destructures the CLI's JSON payload into typed columns:

```typescript
// POST /api/projects/:id/events
// CLI sends: { event_type: "task.status_changed", payload: { sprint, task_num, old_status, new_status } }
// API maps to: INSERT INTO event (event_id, event_type, user_id, project_id, sprint, task_num, old_status, new_status, timestamp) VALUES (...)
```

Each `event_type` has a known set of payload fields. The API validates and maps them. Unknown fields go into `meta`.

### Open questions

- **Local-only events**: Telemetry (sections 3-8) stays in `.pm/workflow.db`. If analytics on tool call patterns is needed later, the API could accept telemetry pushes on `n2o sync` and map them into a separate `telemetry` table in Postgres/ClickHouse.
- **Privacy**: Tool call inputs can contain code snippets. Push tool names + token counts only, or full inputs?
- **Retention**: 30-day retention in Postgres (events are in ClickHouse for long-term). Configurable per org?
